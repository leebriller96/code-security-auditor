# 보안 감사 방법론 (security-audit)

코드 보안 취약점을 **근거 기반**으로 탐지하고, 개발자가 바로 고칠 수 있는 수정 가이드까지 제공하는 방법론입니다.

## 0. 분석 철학

- **방어가 목적**: 취약점을 찾아 고치는 것이 목표. 완성형 익스플로잇은 만들지 않고 PoC는 개념 수준으로만.
- **근거 우선**: 모든 발견 항목은 `파일:라인`과 실제 코드 흐름을 근거로 제시. 추측은 "추정"으로 명시.
- **데이터 흐름 추적**: 외부 입력(source)이 위험한 지점(sink)에 도달하는 경로를 추적하는 것이 핵심.
  - source 예: HTTP 요청 파라미터/헤더/바디, 파일 내용, 환경변수, DB, 외부 API 응답, CLI 인자
  - sink 예: SQL 쿼리, OS 명령 실행, 파일 경로, HTML 렌더링, 역직렬화, 리다이렉트 URL, 템플릿 렌더링

## 1. 코드베이스 인벤토리

분석 전 다음을 파악합니다.

1. 파일 트리와 언어 구성 (`find`, 확장자 분포)
2. 의존성 매니페스트 (`requirements.txt`, `package.json`, `pom.xml`, `build.gradle`, `go.mod` 등)
3. 진입점 (웹 라우트, `main`, 핸들러, 람다, 메시지 컨슈머)
4. 신뢰 경계 (외부 입력이 들어오는 지점)
5. 인증/인가 처리 위치
6. 비밀값 취급 (토큰, 키, 비밀번호) 위치

규모가 크면 위험도 높은 영역부터 우선 분석합니다: 인증·인가 → 입력 처리 → 쿼리/명령 실행 → 파일/네트워크 I/O → 역직렬화 → 암호화.

## 2. 취약점 체크리스트 (OWASP Top 10 + CWE Top 25 기반)

각 항목에 대해 코드에서 해당 패턴을 탐지하고, 발견 시 데이터 흐름으로 검증합니다.

### 인젝션 계열
- **SQL/NoSQL Injection** (CWE-89): 문자열 연결/포맷으로 쿼리 구성. → 파라미터 바인딩 확인.
- **OS Command Injection** (CWE-78): `os.system`, `subprocess(shell=True)`, `exec`, `Runtime.exec`에 외부 입력.
- **Code Injection** (CWE-94): `eval`, `exec`, `Function()`, 동적 import에 외부 입력.
- **LDAP/XPath/Template Injection** (CWE-90/643/1336): SSTI 포함.

### 웹 취약점
- **XSS** (CWE-79): 미이스케이프 출력, `innerHTML`, `dangerouslySetInnerHTML`, 템플릿 raw 출력.
- **SSRF** (CWE-918): 외부 입력으로 구성된 URL에 서버가 요청.
- **Open Redirect** (CWE-601): 외부 입력 기반 리다이렉트.
- **CSRF** (CWE-352): 상태 변경 요청에 토큰/검증 부재.
- **Path Traversal** (CWE-22): 외부 입력이 파일 경로에 직접 사용 (`../` 미차단).

### 인증/인가/세션
- **Broken Auth** (CWE-287/306): 인증 누락, 약한 검증, 인증 우회 경로.
- **Broken Access Control / IDOR** (CWE-639/862): 객체 ID 기반 접근에 소유권 검증 부재.
- **세션 고정/약한 세션** (CWE-384): 예측 가능 토큰, 만료 미설정.

### 암호화/비밀
- **하드코딩된 비밀** (CWE-798): 소스에 API 키/비밀번호/토큰.
- **약한 암호화** (CWE-327): MD5/SHA1 비밀번호 해싱, ECB 모드, 약한 난수(`random` for security).
- **민감정보 평문 저장/전송** (CWE-311/319).

### 역직렬화/파일/메모리
- **안전하지 않은 역직렬화** (CWE-502): `pickle`, `yaml.load`(non-safe), Java 역직렬화, `unserialize`.
- **무제한 파일 업로드** (CWE-434): 확장자/타입/크기 검증 부재.
- **XXE** (CWE-611): 외부 엔티티 허용 XML 파서.

### 설정/기타
- **민감정보 로깅** (CWE-532), **상세 에러 노출** (CWE-209).
- **취약한 의존성** (CWE-1104): 알려진 CVE가 있는 라이브러리 버전.
- **불충분한 입력 검증** (CWE-20), **레이스 컨디션** (CWE-362), **SSRF/오픈 리다이렉트와 결합된 인증 우회**.

## 3. SAST 도구 연동 (hybrid / sast-only 모드)

`tools/run_sast.sh <대상경로>` 가 언어를 감지해 적절한 도구를 실행하고 결과를 JSON으로 모읍니다.

- **Semgrep** (범용, 다언어): 기본 룰셋 `p/default`, 보안 룰셋 `p/security-audit`, `p/owasp-top-ten`.
- **Bandit** (Python): 파이썬 보안 린터.
- **Gitleaks / detect-secrets** (전 언어): 하드코딩된 비밀 탐지.
- **npm audit / pip-audit** (의존성): 알려진 CVE 점검.

SAST 결과 처리 원칙:
- SAST 결과는 **후보**입니다. 각 항목을 코드로 직접 검증해 **오탐을 걸러냅니다.**
- SAST가 보지 못하는 **비즈니스 로직 취약점**(권한 검증 누락, IDOR 등)은 Claude 분석으로 보완합니다.

## 4. 트리아지 (심각도 산정)

심각도 = f(영향도, 악용 가능성, 노출면). CLAUDE.md의 등급표를 기준으로 합니다.

- 외부 미인증 입력으로 즉시 RCE/데이터 유출 → **Critical**
- 인증 필요 또는 조건부지만 심각 → **High**
- 제한적 영향 또는 추가 조건 다수 필요 → **Medium**
- 모범사례 위반/심층 방어 → **Low**
- 취약점 아님(개선 권고) → **Info**

각 항목에 가능하면 CVSS 3.1 벡터를 함께 제시합니다(추정임을 명시).

## 5. 레포트 작성

`templates/report_template.md` 구조를 따릅니다. 각 발견 항목은 반드시 다음을 포함합니다.

1. 제목 + 심각도 배지 + CWE/OWASP 매핑
2. 위치: `파일:라인`
3. 설명: 무엇이 왜 위험한가 (데이터 흐름 포함)
4. 공격 시나리오: 개념 수준 (완성형 익스플로잇 금지)
5. 취약 코드 스니펫 (Before)
6. 수정 방법: 권장 패턴 + 수정 코드 (After), 라이브러리/설정 변경 포함
7. 참고자료: CWE/OWASP/공식 문서 링크

레포트 끝에는 **우선 조치 순서**(remediation roadmap)를 표로 제시합니다.

## 6. 출력물 생성

1. Markdown: `reports/<KST타임스탬프>_security_report.md`
   - 타임스탬프: `TZ=Asia/Seoul date '+%y%m%d%H%M'`
2. HTML + PDF: `python3 tools/build_report.py reports/<...>.md`
   - 같은 디렉토리에 `.html`, `.pdf` 동시 생성.

## 7. 품질 자가 점검 (레포트 제출 전)

- [ ] 모든 발견 항목에 `파일:라인` 근거가 있는가?
- [ ] 오탐을 검증해 걸렀는가? (검토 제외 섹션 기록)
- [ ] 각 항목에 실행 가능한 수정 코드(After)가 있는가?
- [ ] 완성형 익스플로잇이 포함되지 않았는가?
- [ ] 한글로 작성되었는가?
- [ ] 우선 조치 로드맵이 있는가?
